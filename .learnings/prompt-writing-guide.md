# Prompt 写作指南

**来源**: 分析李继刚 ljg-skills 的 Prompt 写作风格
**版本**: v1.0
**日期**: 2026-03-24

---

## 🎯 核心理念

> **Lisp 风格：极简、递归、精确**

---

## 📐 结构模式

### 1. YAML 头部
```yaml
---
name: skill-name
description: "一句话描述 + 触发词"
version: "x.x.x"
---
```

### 2. 标题层级
```
# 主标题（技能名）
## 二级标题（章节）
### 三级标题（子章节）
```

### 3. 代码块规范
- 使用反引号 `code`
- 用 ~code~ 表示内联代码
- 用 =code= 表示内联代码（org-mode）

## ✍️ 写作风格

### 1. 极简动词

| ❌ 避免 | ✅ 使用 |
|--------|--------|
| 请您 | 你 |
| 正在进行 | 正在 |
| 让我们来 | 直接写 |
| 需要注意的是 | 直接写 |
| 非常抱歉 | 不写 |

### 2. 短句优先

- 每行不超过 20 个字
- 一个动作一行
- 换行 = 换意

### 3. 精确触发

描述触发条件要精确：
```
❌ "当用户想要下载东西时"
✅ "用户分享 X/Twitter 链接并想保存媒体"
✅ "用户说 '下载'、'download'、'保存图片'"
```

### 4. 禁止废话

- 不解释为什么这样做
- 不写背景铺垫
- 直接说做什么、怎么做

### 5. 禁止模糊

- ❌ "适当"
- ❌ "可能"
- ❌ "也许"
- ❌ "根据情况"
- ✅ 具体数字、具体条件、具体行为

---

## 🔧 格式规范

### 1. 分隔线
- ❌ `---`（Markdown 分隔线）
- ✅ 空行 + 标题层级

### 2. 列表
- ❌ `* item`（org-mode 中这是标题）
- ✅ `- item` 或 `1. item`

### 3. 加粗
- ❌ `**bold**`
- ✅ `*bold*`

### 4. 代码
- ❌ 反引号内的大段代码用多行代码块
- ✅ 内联用 `~code~` 或 `=code=`

### 5. ASCII Art
允许：`+ - | / \ > < v ^ * = ~ . : # [ ] ( ) _ , ; ! ' "`
禁止：Unicode 绘图字符

## 📋 执行流程写法

### 标准模式

```markdown
## 执行流程

### 1. 解析输入
从用户输入中提取 URL...

### 2. 执行操作
直接用命令...

### 3. 汇报结果
用 ls -lh 列出文件...
```

### 压缩模式

```markdown
### 1. 解析
提取 URL

### 2. 执行
yt-dlp 下载

### 3. 汇报
ls -lh 列出
```

## 🎭 特殊技巧

### 1. 禁止列表
```
禁止：
- xxx
- xxx
- xxx
```
比"不要xxx"更清晰。

### 2. 判断条件
```
判断结果：
- A → 执行 X
- B → 执行 Y
- C → 告知用户
```
比"如果是A就..."更简洁。

### 3. 示例格式
```markdown
## 示例

```
用户: "xxx"
我: "xxx"
```
```

### 4. 故障排除
```markdown
## 故障排除

### 问题1
解决

### 问题2
解决
```

---

## 🚫 常见错误

1. **触发词不精确** → 用户不知道怎么触发
2. **步骤太模糊** → "适当处理"等于没说
3. **格式不统一** → 混用 Markdown 和 org-mode
4. **条件不明确** → "必要时"太模糊
5. **缺少边界** → 没写禁止什么

---

## 💡 顿悟提炼

每次写完 Prompt，问自己：

1. 能删掉哪个字？
2. 能用更短的词吗？
3. 哪句话可以压成一行？
4. 用户能看懂怎么触发吗？
5. 能用一句话说清楚这个 Skill 吗？
