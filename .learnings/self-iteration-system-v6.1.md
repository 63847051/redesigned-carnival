## 🔄 5步迭代流程

### Step 1 — 读日志
回顾本次对话，用 ❌ 标出所有出错节点，用 ✅ 标出做得好的节点。

**检查文件**：
- `memory/YYYY-MM-DD.md`（今日记忆）
- `.learnings/improvements/`（改进记录）
- `.learnings/rules/`（固化规则）

---

### Step 2 — 找模式
不孤立看单次失败，找出反复出现的问题类型。

**输出格式**：
```
【模式识别】
- 模式1：_____（出现 ___ 次）
- 模式2：_____（出现 ___ 次）
```

---

### Step 3 — 定根因（压缩版）

**压缩理念**：用最少的词说清最深的理解。

区分三类问题，对症下药：

**能力问题**（我不会）→ 补充知识、主动澄清
**态度问题**（我会但没认真做）→ 加入强制检查步骤
**流程问题**（步骤设计有缺陷）→ 重新设计执行流程

**压缩输出**：
```
【根因压缩】
问题1 → 根因类型 → 一句话顿悟（以后遇到___时，必须___）
问题2 → 根因类型 → 一句话顿悟
```

---

### Step 4 — 制定措施（压缩版）

**压缩理念**：每条措施 = 一个 Prompt = 一套方法论 = 一句顿悟

每条措施必须具体、可验证、可执行：

```
【改进措施】
□ 措施1：遇到_____时，必须先_____（一句话顿悟）
□ 措施2：每次_____前，先检查_____（一句话顿悟）
```

**压缩原则**：
- 一措施一行为，不贪多
- 可截图验证
- 执行时间 < 30秒

**❌ 无效示例**：
- "下次注意"
- "以后小心"
- "我会记住的"

**✅ 有效示例**：
- "GitHub地址必须用完整URL格式 https://github.com/xxx"
- "每次分配给小新任务时，使用 opencode -m opencode/minimax-m2.5-free run"

---

### Step 5 — 写入规则库
将新措施追加到下方「固化规则库」对应分区，注明日期。

## 💎 压缩心法

源自李继刚 ljg-learn 的"压缩"理念：

### 什么是压缩
- 用一本书的厚度讲一个问题
- 用一个 Prompt 封装一套方法论
- 最后压成一句顿悟

### 怎么压缩
1. **展开**：把这个问题的所有角度都想一遍
2. **找筋骨**：哪几个角度指向同一个深层结构？
3. **一句话**：用最简单的话说最深的理解

### 压缩公式
```
复杂问题 → 多角度分析 → 找共同结构 → 一句顿悟
```

### 顿悟标准
- 能用 5 个字说清楚，就不用 10 个字
- 能用一个比喻说清楚，就不用一段解释
- 能被记住的，才是真正的理解

## 📚 固化规则库

> 迭代后持续追加，禁止删除历史规则（只能标注「已优化」）

### A类 — 能力补丁
格式：日期 | 问题描述 | 规则：遇到___时，必须___

**示例**：
```
2026-03-24 | 微信文章读取失败 | 规则：遇到 mp.weixin.qq.com 链接时，必须使用 read-wechat.py
```

---

### B类 — 态度检查点
格式：日期 | 问题描述 | 规则：每次___前，必须先___

**示例**：
```
2026-03-24 | opencode 模型使用错误 | 规则：每次分配给小新任务前，必须检查是否使用 OpenCode CLI
```

---

### C类 — 流程优化
格式：日期 | 问题描述 | 规则：___任务需先___再___

**示例**：
```
2026-03-24 | 任务分配混乱 | 规则：任务分配需先分析类型→选择合适的Agent→执行→汇报
```

## 💡 关键原则

1. **要具体，不要模糊**
   - ❌ "下次注意"
   - ✅ "每次分配给小新任务时，使用 opencode -m opencode/minimax-m2.5-free run"

2. **不甩锅，先从自身找原因**
   - ❌ "上下文丢失"
   - ❌ "工具限制"
   - ✅ "我没有建立正确的规范"

3. **固化到文件**
   - ❌ 写在嘴巴上
   - ✅ 写在文件里

4. **持续迭代**
   - 每次错误都是学习机会
   - 每次迭代都让系统更稳定

5. **压缩顿悟**
   - ❌ 写一大段分析
   - ✅ 压成一句话顿悟
