## 🎯 实施方案：Skill 隔离规则系统

### 为什么选择这个方案？

1. ✅ **立即可用** - 无需配置 OpenClaw 允许列表
2. ✅ **成本为零** - 继续使用免费模型（70%+ 免费）
3. ✅ **90% 隔离** - 通过规则引擎实现上下文隔离
4. ✅ **可扩展** - 易于添加新角色
5. ✅ **可升级** - 未来可迁移到独立子 Agent

### 核心机制

#### 1. 触发词检测
```javascript
设计关键词: ["设计", "图纸", "平面图", "立面图", "天花", "地面", "排砖", "柜体", "会议室"]
技术关键词: ["代码", "爬虫", "数据", "API", "前端", "脚本", "开发", "编程"]
日志关键词: ["日志", "记录", "工作", "任务", "进度", "统计", "汇总"]
```

#### 2. 角色边界
每个专家有明确的职责范围，越界任务自动转发给主控

#### 3. 模型隔离
- 设计专家: `glmcode/glm-4.7`
- 技术专家: `openrouter/gpt-oss-120b`（免费）
- 小蓝: `glmcode/glm-4.5-air`（免费）

## 🔄 工作流程

```
用户消息
    ↓
大领导（主控 Agent）接收
    ↓
意图识别 → 检测关键词
    ↓
分配给对应专家（带隔离规则）
    ↓
专家在隔离上下文中处理
    ↓
检测越界？→ 是 → 转发给大领导
    ↓ 否
专家汇报成果
    ↓
大领导汇总 → 用户
```

## 🚀 未来升级路径

### Phase 2: 独立子 Agent（可选）
如果需要更强的隔离，可以升级到独立子 Agent：

```javascript
sessions_spawn({
  runtime: "acp",        // ACP 编码会话
  mode: "session",       // 持久化
  thread: true,          // 线程绑定
  model: "...",
  label: "...",
  task: "..."
})
```

**优点**: 100% 隔离、真正独立的进程
**代价**: 需要配置允许列表、稍高的成本

### 何时升级？
- 需要子 Agent 并行处理多个任务
- 需要子 Agent 主动发起任务（不只是响应）
- 需要 100% 的上下文隔离
