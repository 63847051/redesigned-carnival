## 三、深度分析：6 个核心技能

### 3.1 test-driven-development（测试驱动开发）

#### 3.1.1 核心流程

```
┌─────────────────────────────────────────────────────────────┐
│                    TDD 循环 (RED-GREEN-REFACTOR)            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐  │
│   │   RED   │ →  │ GREEN   │ →  │REFACTOR │ →  │ COMMIT  │  │
│   │ 写失败测试│    │ 最小代码│    │  重构   │    │  提交   │  │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘  │
│        ↓              ↓              ↓                       │
│   "测试失败"     "测试通过"     "测试通过"                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.1.2 实现细节

**RED 阶段**:
- 编写一个失败的测试用例
- 验证失败原因：功能不存在，不是测试本身有问题
- 不允许在测试之前编写任何生产代码

**GREEN 阶段**:
- 编写最小代码让测试通过
- 不追求完美，只求通过
- 可以有代码异味，后续 REFACTOR 阶段处理

**REFACTOR 阶段**:
- 重构代码提升质量
- 保持测试通过
- 可以添加新测试继续循环

#### 3.1.3 最佳实践

1. **最小步进**: 一次只写一个测试，一个行为
2. **真实数据**: 使用实际数据，避免过度 mock
3. **强制删除**: 删除测试前写的代码
4. **铁律**: "没有失败测试就不能写生产代码"

### 3.2 systematic-debugging（系统化调试）

#### 3.2.1 四阶段流程

```
┌─────────────────────────────────────────────────────────────┐
│                    系统化调试四阶段                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Phase 1        Phase 2         Phase 3        Phase 4      │
│  ┌──────┐      ┌──────┐       ┌──────┐      ┌──────┐       │
│  │根因  │ →   │模式   │  →    │假设   │ →    │实现   │       │
│  │调查  │     │分析   │       │与测试 │      │修复   │       │
│  └──────┘      └──────┘       └──────┘      └──────┘       │
│                                                             │
│  • 错误信息    • 参考实现      • 单一假设    • 失败测试     │
│  • 复现       • 差异识别      • 最小变更    • 单一修复     │
│  • 最近变更   • 类似代码      • 验证后再继   • 验证修复     │
│  • 证据收集                                   • 3+失败=架构 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.2 实现细节

**Phase 1: 根因调查**
- 仔细阅读错误信息
- 一致地复现问题
- 检查最近变更
- 收集多组件系统证据
- 追踪数据流

**Phase 2: 模式分析**
- 找到类似工作的代码
- 与参考实现比较
- 识别差异点

**Phase 3: 假设与测试**
- 形成单一假设
- 最小化变更测试
- 验证后再继续下一步

**Phase 4: 实现**
- 创建失败测试用例（防止回归）
- 实现单一修复
- 验证修复有效
- 如 3+ 修复失败 → 质疑架构决策

#### 3.2.3 最佳实践

1. **铁律**: "没有根因调查就不能修复"
2. **最小变更**: 每次只测试一个假设
3. **3+ 失败规则**: 多次修复失败意味着架构问题，应停止并讨论

### 3.3 brainstorming（头脑风暴）

#### 3.3.1 核心流程

```
┌─────────────────────────────────────────────────────────────┐
│                      头脑风暴工作流                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   用户需求                                                     │
│       ↓                                                      │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ 1. 探索项目上下文                                      │ │
│   │    - 文件结构、文档、最近提交                          │ │
│   │ 2. 提问理解需求（一次一个）                            │ │
│   │ 3. 提出 2-3 种方案及权衡                              │ │
│   │ 4. 展示设计，分段获得批准                              │ │
│   │ 5. 编写设计文档保存到 docs/superpowers/specs/         │ │
│   │ 6. 自审规格文档                                        │ │
│   │ 7. 用户审查                                            │ │
│   └─────────────────────────────────────────────────────────┘ │
│       ↓                                                      │
│   writing-plans                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.3.2 最佳实践

1. **一次只问一个问题**: 避免一次抛出多个问题
2. **选择题优先**: 优先提供选项而非开放式问题
3. **YAGNI 原则**: 移除不必要的功能
4. **小而明确**: 设计分解为小而明确的单元
5. **规格自审**: 检查占位符、矛盾、模糊点
6. **未获批准不能实现**: 必须在设计批准后才能进入实现阶段

### 3.4 writing-plans（编写计划）

#### 3.4.1 任务分解标准

| 维度 | 标准 |
|------|------|
| **时间** | 每个任务 2-5 分钟 |
| **粒度** | 完整的代码在每个步骤中 |
| **文件** | 精确的文件路径 |
| **验证** | 每个任务都有验证步骤 |
| **依赖** | 明确的任务依赖关系 |

#### 3.4.2 计划文档结构

```markdown
# [Feature Name] Implementation Plan

**Goal:** 一句话目标
**Architecture:** 2-3 句方法
**Tech Stack:** 关键技术栈

---

### Task N: [Component Name]

**Files:**
- Create: exact/path/to/file.py
- Modify: exact/path/to/existing.py:123-145
- Test: tests/exact/path/to/test.py

- [ ] **Step 1: Write failing test**
- [ ] **Step 2: Run test to verify it fails**
- [ ] **Step 3: Write minimal code**
- [ ] **Step 4: Run test to verify it passes**
- [ ] **Step 5: Refactor if needed**
- [ ] **Step 6: Commit**
```

#### 3.4.3 最佳实践

1. **每个步骤必须有实际内容**: 不能有占位符
2. **精确文件路径**: 避免模糊描述
3. **完整代码**: 每个步骤包含完整代码
4. **自审清单**: 规格覆盖、占位符扫描、类型一致性

### 3.5 subagent-driven-development（子代理驱动开发）

#### 3.5.1 两阶段审查机制

```
┌─────────────────────────────────────────────────────────────┐
│                  两阶段审查流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   子代理完成任务                                             │
│       ↓                                                     │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ Phase 1: 规范合规性审查                              │   │
│   │ - 是否完成了所有要求的功能？                          │   │
│   │ - 是否使用了指定的文件路径？                          │   │
│   │ - 是否符合指定的技术栈？                             │   │
│   │ - 是否通过了指定的验证步骤？                          │   │
│   └─────────────────────────────────────────────────────┘   │
│       ↓                                                     │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ Phase 2: 代码质量审查                                │   │
│   │ - 代码可读性、可维护性                               │   │
│   │ - 错误处理、边界情况                                 │   │
│   │ - 性能、安全                                         │   │
│   │ - 测试覆盖                                           │   │
│   └─────────────────────────────────────────────────────┘   │
│       ↓                                                     │
│   关键问题 → 阻塞，必须修复                                  │
│   非关键问题 → 可选修复                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.5.2 最佳实践

1. **两阶段顺序不能颠倒**: 先规范，后质量
2. **永远不要跳过任何审查**
3. **模型选择**: 简单任务用便宜模型，复杂任务用强大模型

---

### 3.6 verification-before-completion（完成前验证）

#### 3.6.1 核心铁律

> **"没有新鲜验证证据就不能声称完成"**

#### 3.6.2 验证流程

```
声称完成
    ↓
识别什么命令证明该声称
    ↓
运行完整命令
    ↓
阅读完整输出，检查退出码
    ↓
验证输出是否确认声称
    ↓
只有这样才能做出声称
```

#### 3.6.3 禁止行为

- ❌ "应该通过"
- ❌ "可能正确"
- ❌ "看起来不错"
- ❌ 在验证前表达满意（"Great!"、"Perfect!"、"Done!"）

#### 3.6.4 最佳实践

1. **证据优先**: 必须有新鲜的验证证据
2. **完整验证**: 运行完整命令，不只检查退出码
3. **代理验证**: 代理声称成功时，独立验证 VCS diff

## 六、改进建议

### 6.1 高优先级（建议立即实施）

#### 6.1.1 新增 systematic-debugging 技能

**原因**: 我们系统缺少系统化调试流程，经常凭直觉修复 bug。

**建议实现**:
- 采用 Superpowers 四阶段调试流程
- 根因调查 → 模式分析 → 假设测试 → 实现修复
- 3+ 次失败应停止并讨论架构

**文件位置**: `/root/.openclaw/workspace/skills/systematic-debugging/SKILL.md`

#### 6.1.2 新增 verification-before-completion 技能

**原因**: 我们系统缺少完成前验证机制，经常过早声称完成。

**建议实现**:
- 要求每个任务完成后必须运行验证命令
- 禁止"应该通过"、"看起来不错"等无证据声称
- 代理声称成功时，独立验证 VCS diff

**文件位置**: `/root/.openclaw/workspace/skills/verification-before-completion/SKILL.md`

### 6.2 中优先级（建议近期实施）

#### 6.2.1 增强触发机制

**原因**: 当前是手动触发，应学习 Superpowers 的上下文感知触发。

**建议实现**:
- 当 Agent 检测到创造性工作时，自动调用 brainstorming
- 当 Agent 检测到 bug 时，自动调用 systematic-debugging
- 任务完成时自动调用 verification-before-completion

#### 6.2.2 增强子代理驱动开发

**原因**: 我们系统没有严格的子代理审查流程。

**建议实现**:
- 引入 subagent-driven-development 的两阶段审查
- Phase 1: 规范合规性
- Phase 2: 代码质量
- 顺序不能颠倒

### 6.3 低优先级（可选实施）

#### 6.3.1 Git Worktree 集成

**适用场景**: 多任务并行时需要隔离工作区。

#### 6.3.2 分支完成工作流

**适用场景**: 复杂的合并/PR 流程需要标准化。

#### 6.3.3 并行代理调度

**适用场景**: 多个独立任务需要并行处理。

## 八、结论

Superpowers 框架提供了 14 个高度组织化的技能，通过强制性的流程确保软件开发质量。核心优势在于：

1. **流程强制**: 100% 强制执行关键流程
2. **证据优先**: 验证后才声称完成
3. **系统化调试**: 根因导向的调试方法
4. **两阶段审查**: 规范合规 + 代码质量

我们的 Multi-Agent 系统在并行效率和灵活性方面有优势，建议通过新增 systematic-debugging 和 verification-before-completion 两个核心技能，并增强触发机制和审查流程，进一步提升系统质量。
