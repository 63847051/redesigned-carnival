# TDD 工作流程文档

## 标题
文档驱动开发（TDD）工作流程 - [日期 YYYY-MM-DD]

## 概述

### 什么是TDD（Test-Driven Development，测试驱动开发）
TDD是一种软件开发方法论，核心思想是"先写测试，再写代码"。通过预先编写测试用例来指导代码开发，确保代码质量和功能完整性。

### 什么是文档驱动开发（TDD - Test-Driven Documentation）
本文档的TDD指的是"Test-Driven Documentation"，即"先写文档，再执行"。在开始任何工作之前，先编写完整的任务执行文档，然后按照文档步骤执行，确保工作流程的规范性和可追溯性。

### 核心原则
1. **文档先行**：任何任务都要先编写完整的执行文档
2. **步骤清晰**：文档必须包含明确的执行步骤
3. **验证完备**：每个步骤都要有相应的验证标准
4. **持续改进**：执行过程中发现的问题要及时更新文档

## TDD工作流程

### 阶段1：需求分析阶段

#### 1.1 明确任务目标
```markdown
## 标题
[任务名称] - [日期 YYYY-MM-DD]

## 描述
**任务目标**：
- [明确的任务目标描述]
- [预期达成的具体成果]

**背景说明**：
- [任务背景和重要性]
- [涉及的相关方或系统]
```

#### 1.2 任务复杂度评估
- **简单任务**（< 5分钟）：直接执行步骤2
- **中等任务**（5-15分钟）：需要详细计划，执行步骤2-4
- **复杂任务**（>15分钟）：需要完整规划，执行所有步骤

#### 1.3 模板选择
根据任务类型选择合适的模板：
- `task-template.md` - 通用任务执行模板
- `config-template.md` - 配置文档模板
- `review-template.md` - 审查检查清单

### 阶段2：文档编写阶段

#### 2.1 创建任务文档
```bash
# 复制模板并修改
cp /root/.openclaw/workspace/templates/docs/task-template.md ./[任务名称]-YYYY-MM-DD.md
```

#### 2.2 填写文档内容
按照模板要求，完整填写以下内容：
- **标题信息**：任务名称、日期、版本
- **任务描述**：目标、背景、范围
- **执行步骤**：准备、执行、验证、交付
- **验证标准**：功能、质量、风险评估
- **交付物**：代码、文档、其他
- **反思总结**：成功经验、改进空间、学习收获

#### 2.3 文档质量检查
```bash
# 使用质量检查脚本验证文档完整性
/root/.openclaw/workspace/scripts/doc-quality-check.sh [任务文档路径]

# 示例
./doc-quality-check.sh ./example-task-2026-03-28.md
```

#### 2.4 文档评审
- **自我评审**：检查文档的逻辑性和完整性
- **团队评审**：如有需要，请相关人员评审文档
- **修改完善**：根据评审意见修改文档

### 阶段3：执行实施阶段

#### 3.1 按文档执行
严格按照编写好的文档步骤执行：
```bash
# 准备阶段
- [ ] 确认需求和范围
- [ ] 收集必要信息
- [ ] 准备工具和环境

# 执行阶段
- [ ] 具体操作步骤 1
- [ ] 具体操作步骤 2
- [ ] 具体操作步骤 3

# 验证阶段
- [ ] 功能测试
- [ ] 性能测试
- [ ] 用户验收测试

# 交付阶段
- [ ] 文档整理
- [ ] 代码提交
- [ ] 结果汇报
```

#### 3.2 记录执行过程
- **执行日志**：记录每个步骤的执行结果
- **问题记录**：记录执行过程中遇到的问题
- **解决方案**：记录问题解决过程

#### 3.3 实时验证
每个步骤完成后，立即进行验证：
```bash
# 验证函数
verify_step() {
    local step_name="$1"
    local verification_command="$2"
    
    echo "验证步骤: $step_name"
    if $verification_command; then
        echo "✅ $step_name 验证通过"
        return 0
    else
        echo "❌ $step_name 验证失败"
        return 1
    fi
}

# 使用示例
verify_step "代码编译" "make build"
verify_step "单元测试" "npm test"
verify_step "服务启动" "systemctl status myservice"
```

### 阶段4：完善归档阶段

#### 4.1 更新文档
根据执行结果，更新原始文档：
- **实际执行情况**：记录实际执行步骤和结果
- **问题解决方案**：添加遇到的问题和解决方案
- **经验教训**：总结成功经验和改进建议

#### 4.2 文档版本管理
```bash
# 文档版本控制
git add [任务文档路径]
git commit -m "任务完成: [任务名称]"
git push
```

#### 4.3 经验沉淀
- **模式提取**：将成功的工作模式提取为模板
- **最佳实践**：总结最佳实践和注意事项
- **流程优化**：根据实际执行优化工作流程

## TDD实施指南

### 适用场景
1. **新功能开发**：任何新功能的开发都应该使用TDD
2. **系统重构**：重构现有系统时使用TDD确保质量
3. **bug修复**：重要的bug修复使用TDD确保完整性
4. **配置管理**：复杂的配置管理使用TDD确保准确性

### 不适用场景
1. **紧急修复**：非常紧急的简单修复
2. **探索性任务**：需要大量探索和试验的任务
3. **学习性任务**：纯粹的学习和研究任务

### 工具支持

#### 文档模板
- **位置**：`/root/.openclaw/workspace/templates/docs/`
- **模板列表**：
  - `task-template.md` - 任务执行模板
  - `config-template.md` - 配置文档模板
  - `review-template.md` - 审查检查清单

#### 质量检查工具
```bash
# 基本用法
./scripts/doc-quality-check.sh

# JSON格式输出
./scripts/doc-quality-check.sh -j

./scripts/doc-quality-check.sh -r "标题|描述|步骤|验证|结果"
```

#### 快速启动脚本
```bash
#!/bin/bash
# TDD 快速启动脚本

# 1. 创建任务文档
create_task_doc() {
    local task_name="$1"
    local template_path="/root/.openclaw/workspace/templates/docs/task-template.md"
    local output_path="./${task_name}-$(date +%Y-%m-%d).md"
    
    cp "$template_path" "$output_path"
    echo "✅ 任务文档已创建: $output_path"
    return 0
}

# 2. 检查文档质量
check_doc_quality() {
    local doc_path="$1"
    /root/.openclaw/workspace/scripts/doc-quality-check.sh -v "$doc_path"
}

# 3. 完成TDD流程
complete_tdd() {
    local task_name="$1"
    
    echo "开始TDD工作流程..."
    create_task_doc "$task_name"
    
    local doc_path="./${task_name}-$(date +%Y-%m-%d).md"
    if [[ -f "$doc_path" ]]; then
        echo "请编辑任务文档: $doc_path"
        check_doc_quality "$doc_path"
        echo "请按照文档执行任务..."
    fi
}

# 使用示例
# complete_tdd "示例任务"
```

### 最佳实践

#### 文档编写原则
1. **简洁明了**：文档要简洁，避免冗长
2. **具体可执行**：步骤要具体，可以直接执行
3. **可验证性**：每个步骤都要有明确的验证标准
4. **完整性**：覆盖任务的全生命周期

#### 执行监控
```bash
# 执行监控脚本
monitor_execution() {
    local doc_path="$1"
    local start_time=$(date +%s)
    
    echo "开始执行任务..."
    echo "开始时间: $(date)"
    
    # 按步骤执行
    while IFS= read -r line; do
        if [[ $line =~ ^\s*-\s*\[ \]\s*(.+)$ ]]; then
            local step="${BASH_REMATCH[1]}"
            echo "执行: $step"
            # 这里可以添加实际的执行逻辑
            # $(execute_step "$step")
            sed -i "s/-\[ \]/-\[x\]/" "$doc_path"
        fi
    done < "$doc_path"
    
    local end_time=$(date +%s)
    local duration=$((end_time - start_time))
    
    echo "任务完成"
    echo "结束时间: $(date)"
    echo "总耗时: $duration 秒"
}

# 使用示例
# monitor_execution "./task-2026-03-28.md"
```

2. **执行者**：按照文档执行任务
3. **验证者**：负责验证执行结果
4. **评审者**：负责评审文档质量

#### 协作流程
1. **文档评审**：执行前评审文档可行性
2. **执行记录**：记录执行过程中的问题和解决方案
3. **结果验证**：多人交叉验证执行结果
4. **经验分享**：定期分享TDD实施经验

## 常见问题与解决方案

### 问题1：文档编写耗时过长
**解决方案**：
- 使用模板快速创建文档
- 重点关注关键步骤
- 利用工具辅助编写

### 问题2：执行过程中偏离文档
**解决方案**：
- 定期检查执行进度
- 及时更新文档内容
- 建立变更管理机制

### 问题3：验证标准不明确
**解决方案**：
- 制定统一的验证标准
- 使用自动化测试工具
- 建立质量检查清单

### 问题4：团队执行不统一
**解决方案**：
- 建立统一的TDD规范
- 定期培训和经验分享
- 使用工具监控执行情况

## 总结

文档驱动开发（TDD）是一种有效的工作方法，能够：
1. **提高工作质量**：通过预先规划和详细步骤
2. **增强可追溯性**：完整记录工作过程
3. **促进知识沉淀**：积累经验和最佳实践
4. **降低沟通成本**：统一的文档格式和流程

通过实施TDD工作流程，可以形成"文档即代码"的良好习惯，从根本上解决"事后补文档"的问题，提高整体工作效率和质量。
