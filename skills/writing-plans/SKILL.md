---

## 📖 概述

详细计划分解技能将复杂任务分解为**小型、可执行、可验证的任务**，每个任务 2-5 分钟完成，包含精确的文件路径、完整的代码、验证步骤。

### 核心原则

1. **小任务原则** - 每个任务 2-5 分钟可完成
2. **精确性原则** - 明确的文件路径和代码
3. **可验证原则** - 每个任务都有验证步骤
4. **依赖关系** - 明确任务间的依赖关系
5. **可并行原则** - 无依赖的任务可并行执行

---

## 🎯 触发条件

**自动触发**：
- 需求澄清后，任务比较复杂（> 15 分钟）
- 涉及多个文件或多个步骤
- 需要多个 Agent 协作

**手动触发**：
- 用户要求"详细计划"
- 大领导认为任务需要分解

## 🔄 工作流程

### 第 1 步：分析任务复杂度

```
任务: "创建一个股票分析系统"

复杂度评估:
- 预计时间: 2-3 小时
- 涉及文件: 10+ 个
- 涉及 Agent: 小新 + 设计专家
- 结论: 需要详细计划
```

### 第 2 步：识别主要组件

```
主要组件:
1. 数据获取模块
2. 数据处理模块
3. 可视化模块
4. 用户界面
5. 配置管理
```

### 第 3 步：分解为小型任务

```
任务分解:

【数据获取模块】
- Task 1: 创建数据获取接口文件 (5 分钟)
  - 文件: `src/api/stock.ts`
  - 代码: 定义 API 接口
  - 验证: TypeScript 编译通过

- Task 2: 实现股票数据获取函数 (5 分钟)
  - 文件: `src/api/stock.ts`
  - 代码: 实现获取函数
  - 验证: 单元测试通过

【数据处理模块】
- Task 3: 创建数据处理类 (5 分钟)
  - 文件: `src/processor/stock-processor.ts`
  - 代码: 定义类结构
  - 验证: TypeScript 编译通过

- Task 4: 实现技术指标计算 (5 分钟)
  - 文件: `src/processor/stock-processor.ts`
  - 代码: 实现计算逻辑
  - 验证: 单元测试通过

...
```

### 第 4 步：确定依赖关系

```
依赖关系图:

Task 1 (创建接口)
  ↓
Task 2 (实现获取) → Task 3 (创建处理器)
                           ↓
                     Task 4 (实现计算)
                           ↓
                     Task 5 (可视化)

可并行:
- Task 1 和 Task 6 可并行（无依赖）
```

### 第 5 步：分配给 Agent

```
任务分配:

小新（技术支持）:
- Task 1: 创建数据获取接口
- Task 2: 实现股票数据获取
- Task 3: 创建数据处理类
- Task 4: 实现技术指标计算

设计专家:
- Task 7: 设计可视化界面
- Task 8: 设计用户交互流程
```

### 第 6 步：执行和验证

```
执行流程:

并行执行（无依赖）:
- 小新执行 Task 1, 2, 3, 4
- 设计专家执行 Task 7, 8

验证每个任务:
- 检查文件是否存在
- 运行 TypeScript 编译
- 运行单元测试
- 代码审查

汇总结果:
- 所有任务完成
- 集成测试
- 生成报告
```

## 💡 最佳实践

### ✅ 推荐做法

1. **任务越小越好**
   - 每个任务 2-5 分钟
   - 容易验证和调试

2. **明确依赖关系**
   - 哪些任务必须先完成
   - 哪些任务可以并行

3. **包含验证步骤**
   - 编译通过
   - 测试通过
   - 代码审查

4. **预留缓冲时间**
   - 每个任务 +20% 时间
   - 应对意外问题

### ❌ 避免做法

1. **任务太大**
   - 超过 10 分钟的任务需要再分解

2. **依赖不明确**
   - 不清楚哪些任务有依赖

3. **没有验证**
   - 无法确认任务是否完成

4. **过于僵化**
   - 计划应该可以调整
