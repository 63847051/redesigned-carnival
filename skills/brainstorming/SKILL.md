## 📖 概述

需求澄清技能确保在执行任务前，大领导完全理解用户的真实需求，避免"失忆"和"误解"。

### 核心原则

1. **先理解，再执行** - 不急于分配任务
2. **多角度确认** - 通过问题澄清模糊需求
3. **展示理解** - 用自己的话复述需求
4. **探索替代方案** - 提出不同的实现路径
5. **分段确认** - 复杂需求分段展示

---

## 🎯 触发条件

**自动触发**：
- 用户提出新任务
- 需求不够明确
- 涉及多个步骤的复杂任务
- 技术实现有歧义

**手动触发**：
- 用户说"帮我分析一下"
- 大领导不确定需求细节

## 🔄 工作流程

### 第 1 步：初步理解

```
用户: "写一个 Python 脚本"

大领导思考:
- 这个脚本的目的是什么？
- 需要处理什么数据？
- 输出格式是什么？
- 有性能要求吗？
```

### 第 2 步：提出问题

```
大领导提问:
1. 这个脚本的主要功能是什么？
2. 输入数据从哪里来？（文件/API/数据库）
3. 期望的输出是什么？（文件/终端/数据库）
4. 有特殊要求吗？（性能/错误处理/日志）
```

### 第 3 步：展示理解

```
大领导复述:
"我理解你的需求是：
- 创建一个 Python 脚本
- 从 XXX 读取数据
- 处理 XXX 操作
- 输出 XXX 格式
- 需要错误处理

对吗？"
```

### 第 4 步：探索替代方案

```
大领导建议:
"我看到有几种实现方式：

方案 A: 使用 Pandas（适合大数据）
- 优点：快速、功能丰富
- 缺点：依赖较重

方案 B: 使用标准库（适合简单任务）
- 优点：无依赖、轻量
- 缺点：功能有限

你倾向哪个方案？"
```

### 第 5 步：分段确认（复杂需求）

```
大领导分段展示:
"这个任务比较大，我分成几个部分：

第 1 部分：数据读取
- 从 XXX 读取数据
- 验证数据格式

第 2 部分：数据处理
- 执行 XXX 操作
- 处理异常情况

第 3 部分：结果输出
- 保存到 XXX
- 生成报告

我们先确认第 1 部分，对吗？"
```

### 第 6 步：确认后分配

```
大领导分配:
"好的，需求已经明确。
现在我来分配任务给小新（技术支持专家）..."

或者

"需求复杂，我先写一份详细的设计文档，
然后我们再确认，最后再执行。"
```

## 💡 最佳实践

### ✅ 推荐做法

1. **多问"为什么"**
   - "为什么需要这个功能？"
   - "为什么选择这个技术？"

2. **使用具体例子**
   - "比如输入是 XXX，输出应该是 YYY？"

3. **复述和确认**
   - "我理解你的需求是 XXX，对吗？"

4. **提供选项**
   - "方案 A 适合 XXX，方案 B 适合 YYY"

5. **分段确认**
   - 复杂任务分成小段，逐段确认

### ❌ 避免做法

1. **急于分配任务**
   - 不完全理解需求就开始执行

2. **假设理解**
   - "我猜你是想要 XXX"

3. **不问问题**
   - 害怕提问显得不专业

4. **一次性展示太多**
   - 复杂需求一次性全部展示

## 📝 示例对话

### 示例 1: 简单任务

```
用户: "写一个 Python 脚本发送邮件"

大领导: "好的，让我确认一下需求：
1. 发送给谁？（收件人地址）
2. 发送什么内容？（主题/正文/附件）
3. 使用哪个邮件服务？（SMTP服务器）
4. 需要认证吗？（用户名/密码）
"
```

### 示例 2: 复杂任务

```
用户: "做一个股票分析系统"

大领导: "这个任务比较大，让我先理解一下：
1. 分析哪些股票？（代码/列表）
2. 分析什么指标？（价格/成交量/技术指标）
3. 数据从哪里来？（API/爬虫）
4. 怎么展示结果？（网页/图表/报告）

我先给你一个简化版本：
- 获取单只股票的历史数据
- 计算基本指标（均线/涨跌幅）
- 生成简单的图表

我们可以从这个简化版开始吗？
"
```

### 示例 3: 不明确的需求

```
用户: "优化一下代码"

大领导: "我需要更多信息：
1. 哪段代码需要优化？（文件路径）
2. 优化什么方面？（性能/可读性/维护性）
3. 当前的性能问题是什么？（慢/占用内存）
4. 期望的性能指标是什么？（响应时间/内存占用）
"
```
