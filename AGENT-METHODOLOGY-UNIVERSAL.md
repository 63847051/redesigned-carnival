# OpenClaw 多 Agent 方法论通用化指南

**基于文章**: 《OpenClaw实现24小时股票分析自由，0 代码保姆级实战教程》
**核心思想**: 从用户到管理者，管理 AI 团队而不是被 AI 管理

---

## 🎯 方法论核心框架

### 四步搭建法（通用版）

```
模糊需求 → 需求拆解 → 逐个搭建并验证 → 串成流水线
```

**核心原则**:
- ✅ **一个人什么都干 = 加班**
- ✅ **四个人各干各的 = 体系**
- ✅ **上一个人的产出 = 下一个人的输入**

---

## 📋 通用 Prompt 模板

### 需求拆解 Prompt（任何场景可用）

```
你是一个 AI Agent 架构师。我想在 OpenClaw 里搭建一个[场景描述]的 Skill 体系。

请帮我完成以下任务：
1. 把我的需求拆解为 3-5 个独立的子任务（每个子任务对应一个 Agent 角色）
2. 为每个子任务明确：
   - 角色名称和职责定义
   - 输入数据来源（API/文件/用户输入）
   - 输出格式和交付物
   - 需要使用的工具或 API
   - 触发条件（什么时候这个 Agent 该开始工作）
3. 设计这些 Agent 之间的协作流程（谁先谁后、谁给谁传数据）
4. 列出需要的外部数据源和 API
5. 给出安全注意事项

我的场景描述：[在这里填入你的具体需求]
```

## 🎯 10+ 应用场景详解

### 1. 竞品监控系统

**场景描述**:
每天自动扫 10 个竞品官网，有新动态了自动总结、自动推送

**Agent 拆解**:
```
1. 信息抓取 Agent（侦察员）
   - 职责: 每天 9:00 扫描竞品官网、博客、社交媒体
   - 输入: 竞品 URL 列表
   - 输出: 原始 HTML/内容（JSON 格式）

2. 变化检测 Agent（对比员）
   - 职责: 对比昨天和今天的数据，找出变化
   - 输入: 两天的新旧数据
   - 输出: 变化清单（新增/修改/删除）

3. 报告汇总 Agent（分析师）
   - 职责: 分析变化的影响，生成简要报告
   - 输入: 变化清单
   - 输出: Markdown 报告

4. 推送通知 Agent（秘书）
   - 职责: 把报告推送到飞书/钉钉/企业微信
   - 输入: 分析报告
   - 输出: 消息通知
```

**协作流程**:
```
信息抓取 → 变化检测 → 报告汇总 → 推送通知
```

## 🔧 关键技术点

### 1. 数据接力（核心）

**原则**: 上一个人的产出 = 下一个人的输入

**实现方式**:
- 文件接力: JSON/Markdown 文件
- 消息接力: OpenClaw 消息传递
- 数据库接力: SQLite/Redis

### 2. description 写法（最重要）

**好的示例**:
```markdown
description: |
  当用户要求获取 A 股行情数据、查看股票价格、获取市场概览时使用此技能。
  不适用于：加密货币数据、美股数据、基金净值查询。
```

**关键点**:
- ✅ 明确适用场景
- ✅ **负面边界定义**（不适用于什么）

### 3. SOUL.md 的重要性

定义 Agent 的"职业操守":
```markdown
# 你是一名 A 股技术分析师

## 身份
- 专注 A 股市场技术面分析
- 分析风格偏保守，注重风险控制

## 禁忌
- 不推荐个股，只做技术面陈述
- 不使用"必涨""稳赚"等误导性词汇
```

## 💡 方法论总结

### 核心思维转变

**从**: 用户 - AI 问答模式
**到**: 管理者 - AI 团队模式

### 四步法（通用版）

1. **需求拆解**: 把大任务拆成 3-5 个小任务
2. **逐个搭建**: 搭一个测一个，确保每个能独立工作
3. **数据接力**: 设计 Agent 之间的数据传递
4. **流程串连**: 配置自动化协作流程

### 成功关键

- ✅ **单一职责**: 一个 Agent 只干一件事
- ✅ **清晰边界**: 明确什么该干、什么不该干
- ✅ **数据接力**: 上一个人产出 = 下一个人输入
- ✅ **逐步验证**: 搭一个测一个，不要一口气搭完

## 🎯 实践建议

### 从小场景开始
- ❌ 不要一上来就搭复杂的系统
- ✅ 从 2 个 Agent 的简单协作开始
- ✅ 验证可行后再扩展

### 复用已有 Skill
- ✅ OpenClaw 社区有很多现成的 Skill
- ✅ 先看有没有能复用的
- ✅ 改改比从头写快

### 持续迭代
- ✅ 第一版不需要完美
- ✅ 跑起来再优化
- ✅ 记录每次迭代的经验
